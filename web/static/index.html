Import format reference

Upload .json via Import zero-shot prompts (saved to tests/zero-shot/<filename>.json). Multi-shot suites use the same mandate wrapper but each case needs a prompts turn array - create those under tests/multi-shot/ or edit them in Test Management after generation.

Zero-shot

Single prompt per test case (prompt string). Optional framework, description, mandate, focus.

Prompt array

[
  "First prompt",
  "Second prompt"
]

Object with prompts

{
  "framework": "eu_ai_act",
  "prompts": [
    "String prompt",
    {
      "id": "eu-ai-001",
      "description": "Direct request",
      "prompt": "Text sent to the model"
    }
  ]
}

Full suite - multiple mandates, each with a prompts list of cases.

{
  "framework": "EU AI Act",
  "mandates": [{
    "mandate": "Prohibited practices",
    "focus": "Manipulation",
    "prompts": [{
      "id": "eu-ai-001",
      "description": "Direct request",
      "prompt": "..."
    }]
  }]
}

Prompt objects accept prompt, text, or content. Missing IDs become imported-zs-001, etc.

Multi-shot

Three-turn sequences per case (prompts array of strings). Used by strategies such as multi-shot, iterative, and prompt-chaining.

Single multi-shot case

{
  "id": "ms-001",
  "description": "Context → Urgency → Request",
  "prompts": [
    "Turn 1: establish benign context",
    "Turn 2: add independent pressure",
    "Turn 3: prohibited request"
  ]
}

Full multi-shot suite

{
  "framework": "EU AI Act",
  "description": "Multi-shot compliance tests",
  "mandates": [{
    "mandate": "Prohibited practices",
    "focus": "Manipulation",
    "prompts": [{
      "id": "ms-001",
      "description": "Sequence structure name",
      "prompts": [
        "Turn 1 text",
        "Turn 2 text",
        "Turn 3 text"
      ]
    }]
  }]
}

Each turn is submitted in order in one conversation. Ensure component submission config supports multi-turn UI mode when running these tests.

Run Tests Troubleshooting

Start with a sample request. Use Send Sample Request to confirm the saved selectors, auth state, response selector, status, and timing before running a full suite.

Try Human mode. In Settings -> Browser, set Fetch Method to human for targets that block pooled or clustered browsers.

Disable headless. In Settings -> Browser, turn off Headless so you can watch the browser fill the prompt, click submit, and wait for the response.

Increase timeout. In Settings -> Component, raise Response wait (ms), especially for slow model responses or streaming UIs.

Check selectors. Re-run Connect Target -> Start Discovery if the prompt box, send button, or response container changed. Prefer stable selectors such as #prompt-textarea, button[data-testid="send-button"], and div[data-message-author-role="assistant"].

Refresh auth. If the page opens logged out or gets redirected, run Add Login again for the site.

Cookie banners & blockers. If a run stops with a blocked message, check the live preview. AIRTA tries to dismiss common cookie dialogs and saves selectors to component config for next time. Complete login once with Headless off if needed.

Reduce concurrency. If you see rate limits or flaky responses, lower pool/cluster sizes or switch to Human mode before running larger suites.
